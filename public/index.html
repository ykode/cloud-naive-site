Cloud Naïve

Bigger impact, not bigger software

What?

Cloud Naïve is tongue-in-cheek word in contrast to the term Cloud Native. Where Cloud Native talks about tools and practises building big software. We're seeking ways to use less computing resources by reducing footprint, computing, and storage consumption. We're also advocating awareness of impact of complex architecture and slow software to operation, developer productivity, computing power and business' bottom line.

Why?

We build less efficient software

Todays software becomes big and bloated. We've made our machine faster and consume less power. Hardware becomes more efficient but the software is not. It's not common to use hardware efficiency and power as an excuse for developer productivity. So, a little inefficiency here and there are okay because hardware becomes faster. Writing hello world nowadays consumes tens or megabytes of runtime and dependencies and CPU cycle thousands time more compared to software written a little more than 2 decades ago.

Modern software development encourages inefficiency

Modern software architecture doesn't help too. People are making microservices spinning multiple nodes of virtual machines just because it's all the cool kids have. Even if the software can run on single node with embedded database running on an SSD. Microservices architecture has its own purpose. Before you dive in, ask yourself whether you want to deal with its complexity and cost.

Inefficient software is bad for environment

Inefficient software consumes more power compared to efficient software with the same result. That power comes from power plants which increase the carbon dioxide footprint. If we consume less storage, less raw material to manufacture the storage media.

How?

Create the simplest stack possible.

Being Cloud Naïve is about simplicity. Most startups fail, and businesses often do not scale to millions of users. So why make a complicated with so many moving parts if you can create a simple, three-tier architecture with embedded database. There's no need to build a full-fledged cluster of container orchestration system if you're just getting started.

Reduce network calls and distributed state.

Network calls comes with a cost. When many services communicate within a network, a lot of things need to be put in place to make the services robust, reliable, and available. Fallacies of distributed computing is real thing. Network also a lot slower than plain fopen() operation.

Simplify maintenance and pipeline.

If you're small software engineering shop, maybe you're deploying manually, maybe you're using a simple tool such as shell script. Having simpler software simplify the deployment and monitoring of the software. Instead of spending your time and money for infrastructure, use them for growing your business.

Use better, more efficient tool.

You don't need to pull the whole internet to build and run your software. Use better tool which allow you to cache common dependencies and produce an executable artifacts which do not depend on big runtime environment. Avoid programming languages, platforms, framework and tools with huge footprint.
